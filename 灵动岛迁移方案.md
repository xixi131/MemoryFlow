# 灵动岛迁移方案

## 1. 目标与范围

本方案用于将 MemoryFlow 当前基于 Electron 的灵动岛窗口，迁移为一个 **macOS 原生本地 App**。

迁移后的目标不是做一个无界面的后台 helper，而是做一个真正符合 macOS 使用习惯的原生应用：

- 顶部居中的灵动岛窗口，与 Mac 刘海区域视觉对齐
- 拥有菜单栏图标、偏好设置、开机启动等原生系统能力
- 保留当前灵动岛已有的核心体验
- 其他现有业务页面和主应用尽量保持不变

本次迁移的范围只覆盖：

- 灵动岛窗口本身
- 灵动岛窗口相关的所有交互和状态
- 灵动岛涉及的音乐接管、提醒、模式切换、认证同步、菜单栏设置

本次迁移 **不要求** 一次性重写整个 MemoryFlow 桌面端。

---

## 2. 最终技术选型

### 2.1 结论

灵动岛迁移后的 mac 端形态，确定为：

- **独立 macOS 原生 App**
- **AppKit + SwiftUI 混合开发**

其中：

- **AppKit** 负责窗口、屏幕、事件、菜单栏、启动项、系统集成
- **SwiftUI** 负责灵动岛内容视图、设置页、菜单栏弹层、状态驱动渲染

### 2.2 为什么不用 Electron 继续做

Electron 在当前 Windows 版中可以完成大部分窗口与前端渲染逻辑，但在 macOS 上，灵动岛需要依赖更深层的系统能力：

- 刘海区域精确定位
- 透明无边框窗口
- 点击穿透与 hover 激活
- 跨显示器重定位
- 菜单栏图标与菜单
- 更符合系统体验的设置入口
- 更自然的音乐接管与权限模型

这些能力在 macOS 上更适合用原生框架实现。继续用 Electron，会变成：

- 窗口壳是 Electron
- 刘海能力需要原生补丁
- 音乐接管需要原生桥接
- 菜单栏和系统行为还要再补平台特化

最终复杂度会高于直接使用原生 App。

### 2.3 为什么不用纯 SwiftUI

SwiftUI 很适合写界面，但灵动岛并不是普通设置页，它强依赖：

- `NSWindow` / `NSPanel`
- 顶部悬浮层级
- 忽略鼠标事件
- 鼠标悬停和穿透切换
- 屏幕 safe area / notch 布局
- 更细粒度的手势和事件控制

这些部分更适合由 AppKit 承担。

因此最合适的不是“纯 AppKit”或“纯 SwiftUI”，而是：

- **AppKit 做系统外壳**
- **SwiftUI 做内容界面**

这也是目前很多复杂 macOS 原生应用常见的组合方式。

## 3. 产品形态定义

迁移后，灵动岛作为一个独立的原生 App 存在，而不是隐藏的辅助进程。

建议产品形态如下：

- 应用启动后，顶部显示灵动岛窗口
- 菜单栏右上角提供一个状态图标
- 菜单栏中可进行基本控制
- 提供原生偏好设置窗口
- 可配置是否开机自动启动
- 可通过菜单栏打开主 MemoryFlow 应用

建议菜单栏包含：

- 显示/隐藏灵动岛
- 切换复习模式 / 待办模式
- 音乐接管开关
- 开机启动开关
- 打开主应用
- 偏好设置
- 退出

## 4. 迁移后的系统边界

## 4.1 迁移策略：只新增，绝对不删除旧代码

本次灵动岛迁移采用严格的增量策略：

- **只新增代码**
- **绝对不删除旧代码**

这是一条强约束，而不是临时建议。

原因如下：

- 当前 Electron 灵动岛仍然是 Windows 端的唯一稳定实现
- 迁移过程中需要持续对照旧行为，作为功能与交互基准
- mac 原生版本在未完全稳定前，不应影响现有桌面端可用性
- 旧代码中仍然包含 Windows 平台继续需要的逻辑，不能在迁移过程中误删

因此，本次迁移的实施原则是：

- 在仓库中新增 `mac-island/` 原生 App 工程
- 原有 Electron 灵动岛代码完整保留
- 原有 React 组件完整保留
- 原有 Windows 音乐接管实现完整保留
- 原有 Electron widget 入口完整保留

迁移期间允许做的事情：

- 新增 mac 原生代码
- 新增平台判断
- 新增桥接层
- 新增配置项
- 新增文档和说明

迁移期间不允许做的事情：

- 删除旧的 Electron 灵动岛组件
- 删除旧的窗口逻辑
- 删除旧的 Windows 音乐接管代码
- 删除任何仅因“未来可能不用”而判断为可删的旧实现

也就是说，在本轮迁移完成之前，旧代码的角色是：

- Windows 平台正式实现
- mac 原生迁移行为对照基线
- 回退和验证保障

只有在未来一个单独的、明确批准的“旧代码退役阶段”中，才允许讨论是否清理历史实现。

### 4.1 保留不动的部分

除本节新增的 mac 原生代码外，以下内容在本轮迁移中默认保持不动：

以下内容尽量保持现状：

- 现有 Electron 主应用
- React 业务页面
- 后端接口和数据结构
- 现有登录、学习、待办、文档等业务流程

### 4.2 迁移出去的部分

以下内容从 Electron 灵动岛迁移到原生 App：

- 灵动岛窗口创建与定位
- 展开/收起/活动态/普通态状态机
- review / todo 模式切换
- 提醒时间自动唤起
- 音乐接管
- hover 和点击穿透
- 指针和触控板手势
- 菜单栏控制
- 偏好设置
- macOS 启动项

### 4.3 原生 App 与主应用的关系

迁移后形成两个桌面端角色：

- **MemoryFlow 主应用**
  - 继续负责完整业务页面
- **MemoryFlow Island 原生 App**
  - 负责灵动岛能力与系统级交互

二者通过轻量桥接通信，但灵动岛 App 要尽量减少对主应用运行状态的依赖。

## 5. 当前代码迁移边界

当前 Windows 版本中，灵动岛主要集中在：

- `front-end/src/components/DynamicIslandWidget.tsx`
- `front-end/electron/main.cjs`
- `front-end/electron/MusicService.cjs`
- `front-end/electron/MusicWorker.cjs`

当前这套实现实际包含的不只是 UI，还包括：

- 音乐模式切换
- review / todo 模式持久化
- 登录态接收
- `/widget/summary`、`/todos/stats`、`/todos/tasks` 等接口轮询
- todo 勾选状态更新
- 外部登录回调
- hover / pointer / trackpad 交互
- 窗口 resize 与 ignore mouse events

因此迁移时不能把它理解成“只搬一个前端组件”，而应该理解成：

**把一个 Electron 小型子系统，迁移成一个原生 App 子系统。**

## 7. 总体架构设计

### 7.1 分层原则

原生 App 采用四层结构：

1. **System Layer**
   - AppKit 窗口、屏幕、菜单栏、权限、启动项

2. **State Layer**
   - 灵动岛状态机
   - 模式切换
   - 提醒调度

3. **Service Layer**
   - 接口请求
   - token 持久化
   - 音乐接管
   - 主应用桥接

4. **UI Layer**
   - SwiftUI 界面
   - 动画
   - 内容布局

### 7.2 推荐状态模型

建议核心状态统一由 `WidgetStateMachine` 管理：

- `mode`
  - `app`
  - `music`

- `appDisplayMode`
  - `review`
  - `todo`

- `visualState`
  - `collapsed`
  - `activity`
  - `expanded`

- `authState`
  - `loggedOut`
  - `loggedIn`

- `musicState`
  - `inactive`
  - `playing`
  - `paused`

- `hoverState`
  - `idle`
  - `hovered`

这样可以避免把视图逻辑、接口逻辑、手势逻辑混写在一起。

## 8. 音乐接管设计

### 8.1 迁移原则

Windows 当前音乐接管依赖 SMTC，这条路径不能直接迁移到 macOS。

macOS 版本需要重构成 **Provider 架构**，而不是复制 Windows 实现。

### 8.2 推荐设计

定义统一协议：

- `MusicProvider`

统一输出：

- `title`
- `artist`
- `cover`
- `position`
- `duration`
- `status`
- `capabilities`

初期建议 Provider：

- `AppleMusicProvider`
- `SpotifyProvider`
- `GenericMediaKeyProvider`

后续按价值再扩展：

- `QQMusicProvider`
- `NeteaseProvider`

### 8.3 现实目标

一期不要定义成“接管任何音乐软件的完整信息”。

更合理的目标是：

- 主流播放器优先支持
- 未适配播放器优雅降级
- 至少保证媒体键级别的控制能力

## 9. 认证与数据同步策略

### 9.1 原则

原生灵动岛 App 不依赖 Electron 主应用存活，也不依赖浏览器 localStorage。

### 9.2 推荐方案

使用：

- `Keychain` 保存 `accessToken`
- `Keychain` 保存 `refreshToken`
- 本地保存 `expiresAt`

接口层继续复用现有服务端协议：

- `/widget/summary`
- `/todos/stats`
- `/todos/tasks`
- `/todos/tasks/:id/status`
- `/auth/me`
- `/auth/refresh`

### 9.3 与主应用同步

建议原生 App 成为 macOS 桌面端的认证持有方，主应用从原生 App 或共享存储读取登录态。

原因：

- 灵动岛 App 更可能常驻
- 菜单栏与系统级能力都在原生 App 中
- 避免 Electron 和原生各自持有一套独立登录状态

## 11. 完整开发路线大纲

### 11.1 执行总原则

macOS 原生迁移必须以现有 Windows/Electron 灵动岛为唯一行为和视觉基线，不能凭感觉重新设计。

当前基线入口：

- 主体 UI、状态、动画、尺寸和液态连接：`front-end/src/components/DynamicIslandWidget.tsx`
- Electron 窗口创建、置顶区域、resize、点击穿透、菜单模式切换：`front-end/electron/main.cjs`
- Windows 音乐接管：`front-end/electron/MusicService.cjs`、`front-end/electron/MusicWorker.cjs`

允许原生侧重新建模的只有平台能力：

- macOS 屏幕、刘海、菜单栏、外接显示器、分辨率和缩放
- `NSPanel` / `NSWindow` 生命周期
- macOS 点击穿透、hover 监控、触控板事件
- macOS 音乐 Provider、媒体键、权限和 Keychain

不允许自主发挥的内容：

- 灵动岛基础尺寸、活动态尺寸、展开态尺寸
- 左右耳朵液态连接参数和变化规律
- 普通态 / 活动态 / 展开态的状态切换语义
- app / music / review / todo 的优先级和展示内容
- 音乐活动态左封面、右音波的内容结构
- hover 放大、阴影、点击穿透恢复、手势阈值的用户可见行为

重要约束：

- 不能把 `DynamicIslandWidget.tsx` 逐行翻译成一个巨大 SwiftUI 文件。
- 必须先从现有代码抽取“视觉 token + 状态机 + 数据契约 + 交互契约”，再分层迁移。
- 每个 Phase 都必须能追溯到现有源码中的具体常量、函数、分支或 IPC。
- 每个视觉验收都必须包含截图或录屏对比，不接受“看起来差不多”。

### Phase 1：建立原生 App 骨架，但不实现真实灵动岛形态

目标：

- 建立 `mac-island/` 原生工程和可运行壳。
- 只验证 App 生命周期、菜单栏、设置页、透明窗口能力，不在这一阶段调视觉细节。

内容：

- Xcode 工程。
- AppKit + SwiftUI 混合结构。
- `SceneCoordinator` 管理 App 生命周期。
- `StatusBarController` 管理菜单栏入口。
- `PreferencesWindowController` 管理设置页。
- `IslandWindowController` 只负责创建透明 `NSPanel`。
- `IslandRootView` 只显示明确标注的 debug 占位，不作为最终形态。

验收：

- 原生 App 能启动。
- 菜单栏图标可安装和卸载。
- 透明无边框窗口能显示和隐藏。
- 不允许在本 Phase 声称“刘海贴合完成”或“样式完成”。

---

### Phase 2：屏幕、刘海、菜单栏和外接显示器自适应系统

目标：

- 先解决坐标系统，再画灵动岛。
- 所有尺寸必须按当前显示器的 top band 计算，不能硬编码 `y=0` 或固定高度。
- 同时支持内置刘海屏、非刘海屏、外接显示器、多显示器切换、分辨率和缩放变化。

必须建立的原生模型：

- `DisplayMetrics`
  - `screen.frame`
  - `screen.visibleFrame`
  - `screen.safeAreaInsets`
  - `screen.backingScaleFactor`
  - `screen.auxiliaryTopLeftArea`
  - `screen.auxiliaryTopRightArea`
  - `displayID`
- `TopAttachmentMetrics`
  - `kind: notch | menuBar | flatTopFallback`
  - `topBandFrame`
  - `notchFrame`
  - `menuBarHeight`
  - `safeTopInset`
  - `pixelScale`
  - `availableTopWidth`
  - `centerX`
- `IslandWindowFrame`
  - `visibleFrame`
  - `shadowFrame`
  - `hoverHitFrame`
  - `clickThroughFrame`

刘海屏规则：

- 如果 `auxiliaryTopLeftArea` 和 `auxiliaryTopRightArea` 存在并形成中间 gap，则以真实 notch gap 作为 collapsed 普通态贴合基准。
- collapsed 普通态的黑色主体必须完全覆盖真实刘海，不允许露出背景色、白边或半透明边。
- 活动态和展开态允许从 notch 向左右扩展，但顶部连接必须仍然贴住 top edge。
- 展开态高度向下增长，不能向上离开屏幕顶边。

非刘海 / 外接显示器规则：

- 不能继续照搬刘海屏尺寸。
- 必须以当前显示器的菜单栏高度或顶部可用区域作为 top band。
- `menuBarHeight = screen.frame.maxY - screen.visibleFrame.maxY` 只能作为候选值，必须结合当前显示器是否显示菜单栏校验。
- 如果外接显示器没有刘海，collapsed 普通态应进入“top pill fallback”或“menu-bar attached fallback”，而不是假装存在刘海。
- fallback 形态仍然引用 Windows 的 collapsed/activity/expanded 尺寸比例，但要按当前 top band 做缩放和约束。
- 不能让灵动岛超过菜单栏高度后看起来像挂在菜单栏外，除非进入 activity 或 expanded 并有明确动画状态。

缩放策略：

- 以 Windows 设计基准 `36px` collapsed height 作为 design unit。
- 每个显示器计算 `topBandHeight`。
- 建议引入 `visualScale = clamp(topBandHeight / 36, 0.78, 1.18)`。
- collapsed 普通态高度优先贴合 `topBandHeight`。
- activity / expanded 使用 Windows 原始比例，但必须由 `TopAttachmentMetrics` 约束顶部锚点和 shadow 裁切。
- retina 下所有 frame 最终要对齐物理像素：
  - `round(value * backingScaleFactor) / backingScaleFactor`

调试能力：

- 增加仅开发期可打开的 debug overlay 或日志：
  - 当前 displayID
  - screen frame
  - visible frame
  - safeAreaInsets
  - notchFrame
  - menuBarHeight
  - topBandFrame
  - visible island frame
  - shadow frame
  - hover hit frame

验收：

- 内置刘海屏 collapsed 普通态黑色主体严丝合缝覆盖刘海，无白边、无透明边、无一像素错位。
- 外接显示器不超出菜单栏或顶部可用区域，fallback 形态清晰可见。
- 拖动到不同显示器后重新计算 top band。
- 切换缩放、分辨率、休眠恢复后位置正确。
- 截图验收必须覆盖：
  - 内置刘海屏
  - 外接非刘海屏
  - 主显示器切换
  - 菜单栏自动隐藏打开和关闭

### Phase 4：窗口尺寸编排、内容驱动宽度和动画基础设施

目标：

- 在 Phase 3 几何形状之上，建立 Alcove / iPhone Dynamic Island 质感所需的 sizing + motion 基础设施。
- 不再只追求 Windows 行为逐项搬运；Windows 参数作为基线，最终验收以 Mac 刘海贴合、Alcove 极简感、iPhone 弹性动画为准。
- 迁移 Windows 版“固定大窗口 + 内部视觉尺寸变化 + shadow buffer”的策略，但改成 macOS 可自适应窗口编排。

Windows 基线：

- `WINDOW_WIDTH = 620`
- `SHADOW_BUFFER = 120`
- collapsed 时 Electron 窗口仍保持 `height = 300`
- expanded 时窗口高度为视觉高度加 shadow buffer
- renderer 通过 `resize-widget` 通知 main 进程

macOS 原生策略：

- 不再固定 `620 x 300`。
- 引入或收敛现有 `IslandShapeEngine` / `IslandWindowSizingEngine` 边界：
  - 输入：`IslandVisualState`、`TopAttachmentMetrics`、`IslandVisualTokens`、`IslandWidthConstraints`
  - 输出：`visibleFrame`、`shadowFrame`、`contentFrame`、`hitTestFrame`
- collapsed 普通态：
  - visible frame 贴合 top band 或 notch。
  - shadow frame 可向下和左右扩展，但不能在顶部留下透明缝。
- activity：
  - 宽度不能再只按固定 `ACTIVITY_COLLAPSED_WIDTH` 计算。
  - 最终宽度必须由 `max(tokenFallbackWidth, notchBaseWidth + leadingContentWidth + trailingContentWidth + horizontalPadding * 2)` 推导。
  - 当前 preview-only `IslandContentWidthRequirement` 是占位，后续真实内容模块必须替换为测量或声明值。
  - 高度仍以 top band / 36 基准缩放。
- expanded：
  - 宽度按 `expandedWidth = 460` 乘以 screen scale。
  - 高度按 music/app 分支选择 `210 / 320`。
  - 阴影 buffer 必须向下扩，且 blur 衰减尾部不能被窗口边界裁切。
  - 展开态阴影要与左右侧一致，轻、散、无硬边界，不得出现底部灰色块。

动画基础设施要求：

- 引入 `IslandMotionEngine` 或等价 motion profile 层，统一管理 shell frame、path morph、shadow、content opacity/blur 的动画意图。
- 支持可中断动画：用户快速点击、hover、手势打断时，动画从当前 presentation value 继续，不能跳帧或回到旧起点。
- 优先使用 Core Animation / SwiftUI spring / display link 中能稳定达到 60fps/120fps 的方案；复杂路径 morph 不得阻塞主线程。
- Windows `containerSpring` 作为第一组物理参数：
  - stiffness `280`
  - damping `30`
  - mass `1.2`
- 迁移活动态开合动画：
  - `ACTIVITY_COLLAPSE_DURATION_SECONDS = 0.85`
  - `ACTIVITY_OPEN_DURATION_SECONDS = 0.56`
  - `ACTIVITY_SEGMENTED_TIMES = [0, 0.45, 0.55, 1]`
- 迁移 collapsed -> activity 的分段宽度：
  - 中间态 `collapseMidWidth = 155`
  - 高度阶段保持 `36`
- macOS 可以用 `NSAnimationContext`、SwiftUI spring 或自定义 display link，但必须能调出接近 iPhone 灵动岛的弹性：展开有轻微 overshoot，收起有吸附感，activity 进入不僵硬。
- 动画不只改 frame；液态连接、圆角、shadow、内容 fade/blur 要同一条时间线协调。

验收：

- 所有窗口尺寸变化都有 `IslandWindowSizingEngine` 输出日志。
- hover、activity、expanded 不会把窗口推离顶部锚点。
- 外接显示器不会因为固定窗口高度导致越过菜单栏。
- 阴影只在 Windows 基线定义的 hover/expanded 状态出现。
- activity 宽度能随内容需求变化，不被原始刘海遮挡，也不依赖写死宽度。
- compact -> activity -> expanded -> collapsed 连续快速切换无跳变、无硬切、无内容提前弹出。
- 至少生成一组动画验收证据：关键状态视频/GIF 或逐帧截图，检查是否有卡顿、阴影裁切、窗口边界、形状断层。

### Phase 9：交互实测校准、动画性能和视觉验收

目标：

- 在 Phase 5 已建立交互意图和 mock 动画之后，接入真实 macOS 输入事件并做实测校准。
- 目标不是仅仅“能点能滑”，而是接近 iPhone Dynamic Island 的自然反馈：hover、tap、swipe、expanded 收起都要有弹性和可中断性。

Windows 基线：

- hover enter：
  - `setIsHovered(true)`
  - `set-ignore-mouse-events false`
  - collapsed scale `1.06`
  - shadow 出现
- hover leave：
  - `setIsHovered(false)`
  - 非 expanded 且无 pointer gesture 时恢复 click-through
- pointer tap：
  - `TAP_THRESHOLD = 10`
  - tap 切 expanded
- 横向滑动：
  - 根据 `GESTURE_SWITCH_THRESHOLD` 切 forceCompactMode 或 mode。
- 触控板：
  - 横向用于音乐上一首/下一首
  - 纵向上滑关闭/收起
  - 纵向下滑打开/展开
  - gesture lock 和 cooldown 必须保留
- blur：
  - 发送 `widget-collapse`

macOS 实现要求：

- 不允许强行移动用户鼠标位置。
- 刘海区域不可见 cursor 的问题只能通过 hover hit area、窗口层级、cursor rect 或交互区域设计解决，不能用瞬移鼠标解决。
- click-through 必须与 hover / pointer capture / expanded 状态绑定，不能简单开关。
- hover hit frame 必须来自 Phase 4 的 sizing engine。
- 所有手势阈值先从 Windows 记录值开始，再根据 macOS trackpad 实测校准。
- 真实输入事件必须转换成 Phase 5 的 `IslandInteractionIntent`，不能直接绕过状态机改 UI。
- 动画性能目标：
  - 日常路径稳定 60fps。
  - ProMotion 设备上尽量接近 120fps。
  - 快速重复点击、连续滑动、hover 抖动时不出现窗口跳位、状态错乱、路径撕裂。

验收：

- 鼠标移入不会跳走。
- hover 放大和阴影出现。
- hover 退出后 click-through 恢复。
- expanded 外部点击或失焦可收起。
- 手势不会造成 forceCompact、expanded、mode 三者状态错乱。
- 至少保存一组真实设备视频或逐帧截图，覆盖 compact/activity/expanded/hover/music activity 的进入、退出和打断。

## 12. 必须优先锁定的技术决策

在正式开工前，建议先锁定以下 5 个决策：

1. **运行形态**
   - 确认灵动岛是独立原生 App，而不是隐藏 helper

2. **框架组合**
   - 确认采用 AppKit + SwiftUI 混合开发

3. **认证归属**
   - 确认由原生 App 持有桌面端 token

4. **音乐目标**
   - 确认一期目标是主流播放器优先，不承诺所有播放器完整接管

5. **桥接协议**
   - 确认开发期使用 loopback HTTP 还是本地 socket

## 13. 风险点

### 13.1 风险一：脱离现有 Windows 代码自主发挥

这是当前迁移最大的风险。

现有灵动岛的真实产品形态、尺寸、活动态、展开态、音乐态、review/todo 模式、液态连接参数和交互语义都已经在 `DynamicIslandWidget.tsx` 中存在。原生迁移如果绕开这些源码重新设计，就会不断出现形态不一致、状态缺失、屏幕适配错误和动画割裂。

应对：

- Phase 0 先建立源码索引和视觉 token 表。
- 任何视觉、状态、交互实现都必须能回链到现有源码。
- 没有源码依据的形态和参数不得进入原生实现。
- 只能在平台能力层做 macOS 特化，例如屏幕 top band、刘海坐标、Provider、Keychain。

### 13.2 风险二：把原组件逐行翻译成 Swift

这同样危险，但它和“自主发挥”是两种不同问题。

当前 Electron 组件已经把窗口控制、接口、认证、提醒、交互、动画、音乐全堆在一个文件里。逐行翻译会把大文件问题原封不动带到新架构里，后续屏幕适配、Provider、状态机和动画测试都会很难维护。

应对：

- 先从源码抽取视觉 token、路径函数、状态转移和数据契约。
- 再拆 `IslandShapeEngine`、`IslandWindowSizingEngine`、`IslandStateMachine`、业务 service、Provider。
- 最后由 SwiftUI/AppKit 组合渲染。

### 13.3 风险三：屏幕自适应被硬编码破坏

如果继续使用固定高度、固定 `y=0`、固定窗口大小，内置刘海屏、外接显示器、不同缩放比例和菜单栏高度都会出现错位。

应对：

- Phase 2 必须先完成 `TopAttachmentMetrics`。
- 所有 visibleFrame、shadowFrame、hitFrame 都由 sizing engine 计算。
- 每次窗口和视觉改动都跑屏幕矩阵截图。

### 13.4 风险四：音乐接管目标过高

如果一期目标写成“兼容所有音乐软件”，会导致技术风险失控。

应对：

- Provider 架构
- 主流播放器优先
- 未适配播放器降级

### 13.5 风险五：窗口系统和视觉系统分离过晚

如果太晚才处理 notch、hover、click-through、display change、shadow buffer、hit frame 和视觉路径之间的关系，这些底层问题会反复返工。

应对：

- Phase 2 先解决屏幕和 top band。
- Phase 3 迁移视觉几何。
- Phase 4 才把窗口尺寸和动画容器合起来。

### 13.6 风险六：登录状态双写

如果 Electron 和原生各自维护 token，后面很容易状态不同步。

应对：

- 统一 token 持有者
- 统一刷新策略

## 15. 最终建议

本项目的最佳迁移路线不是：

- 继续用 Electron 硬适配 mac 灵动岛
- 或把灵动岛做成没有界面的 helper

而是：

- **把灵动岛做成独立 macOS 原生 App**
- **使用 AppKit + SwiftUI 混合开发**
- **让它成为一个完整、可见、可配置、符合系统习惯的 mac 产品组件**

这样做的优势是：

- 更符合 macOS 用户预期
- 菜单栏、设置页、启动项都天然成立
- 刘海定位和窗口行为更可控
- 音乐接管和系统权限更容易按平台正确建模
- 其他现有业务页面可以继续保留，不需要整站重写

这是当前最稳、也最长期可维护的路线。
