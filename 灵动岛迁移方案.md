# 灵动岛迁移方案

## 1. 目标与范围

本方案用于将 MemoryFlow 当前基于 Electron 的灵动岛窗口，迁移为一个 **macOS 原生本地 App**。

迁移后的目标不是做一个无界面的后台 helper，而是做一个真正符合 macOS 使用习惯的原生应用：

- 顶部居中的灵动岛窗口，与 Mac 刘海区域视觉对齐
- 拥有菜单栏图标、偏好设置、开机启动等原生系统能力
- 保留当前灵动岛已有的核心体验
- 其他现有业务页面和主应用尽量保持不变

本次迁移的范围只覆盖：

- 灵动岛窗口本身
- 灵动岛窗口相关的所有交互和状态
- 灵动岛涉及的音乐接管、提醒、模式切换、认证同步、菜单栏设置

本次迁移 **不要求** 一次性重写整个 MemoryFlow 桌面端。

---

## 2. 最终技术选型

### 2.1 结论

灵动岛迁移后的 mac 端形态，确定为：

- **独立 macOS 原生 App**
- **AppKit + SwiftUI 混合开发**

其中：

- **AppKit** 负责窗口、屏幕、事件、菜单栏、启动项、系统集成
- **SwiftUI** 负责灵动岛内容视图、设置页、菜单栏弹层、状态驱动渲染

### 2.2 为什么不用 Electron 继续做

Electron 在当前 Windows 版中可以完成大部分窗口与前端渲染逻辑，但在 macOS 上，灵动岛需要依赖更深层的系统能力：

- 刘海区域精确定位
- 透明无边框窗口
- 点击穿透与 hover 激活
- 跨显示器重定位
- 菜单栏图标与菜单
- 更符合系统体验的设置入口
- 更自然的音乐接管与权限模型

这些能力在 macOS 上更适合用原生框架实现。继续用 Electron，会变成：

- 窗口壳是 Electron
- 刘海能力需要原生补丁
- 音乐接管需要原生桥接
- 菜单栏和系统行为还要再补平台特化

最终复杂度会高于直接使用原生 App。

### 2.3 为什么不用纯 SwiftUI

SwiftUI 很适合写界面，但灵动岛并不是普通设置页，它强依赖：

- `NSWindow` / `NSPanel`
- 顶部悬浮层级
- 忽略鼠标事件
- 鼠标悬停和穿透切换
- 屏幕 safe area / notch 布局
- 更细粒度的手势和事件控制

这些部分更适合由 AppKit 承担。

因此最合适的不是“纯 AppKit”或“纯 SwiftUI”，而是：

- **AppKit 做系统外壳**
- **SwiftUI 做内容界面**

这也是目前很多复杂 macOS 原生应用常见的组合方式。

## 3. 产品形态定义

迁移后，灵动岛作为一个独立的原生 App 存在，而不是隐藏的辅助进程。

建议产品形态如下：

- 应用启动后，顶部显示灵动岛窗口
- 菜单栏右上角提供一个状态图标
- 菜单栏中可进行基本控制
- 提供原生偏好设置窗口
- 可配置是否开机自动启动
- 可通过菜单栏打开主 MemoryFlow 应用

建议菜单栏包含：

- 显示/隐藏灵动岛
- 切换复习模式 / 待办模式
- 音乐接管开关
- 开机启动开关
- 打开主应用
- 偏好设置
- 退出

## 4. 迁移后的系统边界

## 4.1 迁移策略：只新增，绝对不删除旧代码

本次灵动岛迁移采用严格的增量策略：

- **只新增代码**
- **绝对不删除旧代码**

这是一条强约束，而不是临时建议。

原因如下：

- 当前 Electron 灵动岛仍然是 Windows 端的唯一稳定实现
- 迁移过程中需要持续对照旧行为，作为功能与交互基准
- mac 原生版本在未完全稳定前，不应影响现有桌面端可用性
- 旧代码中仍然包含 Windows 平台继续需要的逻辑，不能在迁移过程中误删

因此，本次迁移的实施原则是：

- 在仓库中新增 `mac-island/` 原生 App 工程
- 原有 Electron 灵动岛代码完整保留
- 原有 React 组件完整保留
- 原有 Windows 音乐接管实现完整保留
- 原有 Electron widget 入口完整保留

迁移期间允许做的事情：

- 新增 mac 原生代码
- 新增平台判断
- 新增桥接层
- 新增配置项
- 新增文档和说明

迁移期间不允许做的事情：

- 删除旧的 Electron 灵动岛组件
- 删除旧的窗口逻辑
- 删除旧的 Windows 音乐接管代码
- 删除任何仅因“未来可能不用”而判断为可删的旧实现

也就是说，在本轮迁移完成之前，旧代码的角色是：

- Windows 平台正式实现
- mac 原生迁移行为对照基线
- 回退和验证保障

只有在未来一个单独的、明确批准的“旧代码退役阶段”中，才允许讨论是否清理历史实现。

### 4.1 保留不动的部分

除本节新增的 mac 原生代码外，以下内容在本轮迁移中默认保持不动：

以下内容尽量保持现状：

- 现有 Electron 主应用
- React 业务页面
- 后端接口和数据结构
- 现有登录、学习、待办、文档等业务流程

### 4.2 迁移出去的部分

以下内容从 Electron 灵动岛迁移到原生 App：

- 灵动岛窗口创建与定位
- 展开/收起/活动态/普通态状态机
- review / todo 模式切换
- 提醒时间自动唤起
- 音乐接管
- hover 和点击穿透
- 指针和触控板手势
- 菜单栏控制
- 偏好设置
- macOS 启动项

### 4.3 原生 App 与主应用的关系

迁移后形成两个桌面端角色：

- **MemoryFlow 主应用**
  - 继续负责完整业务页面
- **MemoryFlow Island 原生 App**
  - 负责灵动岛能力与系统级交互

二者通过轻量桥接通信，但灵动岛 App 要尽量减少对主应用运行状态的依赖。

## 5. 当前代码迁移边界

当前 Windows 版本中，灵动岛主要集中在：

- `front-end/src/components/DynamicIslandWidget.tsx`
- `front-end/electron/main.cjs`
- `front-end/electron/MusicService.cjs`
- `front-end/electron/MusicWorker.cjs`

当前这套实现实际包含的不只是 UI，还包括：

- 音乐模式切换
- review / todo 模式持久化
- 登录态接收
- `/widget/summary`、`/todos/stats`、`/todos/tasks` 等接口轮询
- todo 勾选状态更新
- 外部登录回调
- hover / pointer / trackpad 交互
- 窗口 resize 与 ignore mouse events

因此迁移时不能把它理解成“只搬一个前端组件”，而应该理解成：

**把一个 Electron 小型子系统，迁移成一个原生 App 子系统。**

## 7. 总体架构设计

### 7.1 分层原则

原生 App 采用四层结构：

1. **System Layer**
   - AppKit 窗口、屏幕、菜单栏、权限、启动项

2. **State Layer**
   - 灵动岛状态机
   - 模式切换
   - 提醒调度

3. **Service Layer**
   - 接口请求
   - token 持久化
   - 音乐接管
   - 主应用桥接

4. **UI Layer**
   - SwiftUI 界面
   - 动画
   - 内容布局

### 7.2 推荐状态模型

建议核心状态统一由 `WidgetStateMachine` 管理：

- `mode`
  - `app`
  - `music`

- `appDisplayMode`
  - `review`
  - `todo`

- `visualState`
  - `collapsed`
  - `activity`
  - `expanded`

- `authState`
  - `loggedOut`
  - `loggedIn`

- `musicState`
  - `inactive`
  - `playing`
  - `paused`

- `hoverState`
  - `idle`
  - `hovered`

这样可以避免把视图逻辑、接口逻辑、手势逻辑混写在一起。

## 8. 音乐接管设计

### 8.1 迁移原则

Windows 当前音乐接管依赖 SMTC，这条路径不能直接迁移到 macOS。

macOS 版本需要重构成 **Provider 架构**，而不是复制 Windows 实现。

### 8.2 推荐设计

定义统一协议：

- `MusicProvider`

统一输出：

- `title`
- `artist`
- `cover`
- `position`
- `duration`
- `status`
- `capabilities`

初期建议 Provider：

- `AppleMusicProvider`
- `SpotifyProvider`
- `GenericMediaKeyProvider`

后续按价值再扩展：

- `QQMusicProvider`
- `NeteaseProvider`

### 8.3 现实目标

一期不要定义成“接管任何音乐软件的完整信息”。

更合理的目标是：

- 主流播放器优先支持
- 未适配播放器优雅降级
- 至少保证媒体键级别的控制能力

## 9. 认证与数据同步策略

### 9.1 原则

原生灵动岛 App 不依赖 Electron 主应用存活，也不依赖浏览器 localStorage。

### 9.2 推荐方案

使用：

- `Keychain` 保存 `accessToken`
- `Keychain` 保存 `refreshToken`
- 本地保存 `expiresAt`

接口层继续复用现有服务端协议：

- `/widget/summary`
- `/todos/stats`
- `/todos/tasks`
- `/todos/tasks/:id/status`
- `/auth/me`
- `/auth/refresh`

### 9.3 与主应用同步

建议原生 App 成为 macOS 桌面端的认证持有方，主应用从原生 App 或共享存储读取登录态。

原因：

- 灵动岛 App 更可能常驻
- 菜单栏与系统级能力都在原生 App 中
- 避免 Electron 和原生各自持有一套独立登录状态

### 10.1 菜单栏职责

菜单栏不是附加项，而是原生 App 的重要入口。

菜单栏负责：

- 快速开关灵动岛
- 模式切换

- 主应用唤起

- 音乐接管开关

### 10.2 设置页建议

建议设置页一期包含：

- 启动时自动显示灵动岛
- 开机启动
- 默认显示模式
- 音乐接管开关
- 提醒策略
- 打开主应用方式

## 12. 必须优先锁定的技术决策

在正式开工前，建议先锁定以下 5 个决策：

1. **运行形态**
   - 确认灵动岛是独立原生 App，而不是隐藏 helper

2. **框架组合**
   - 确认采用 AppKit + SwiftUI 混合开发

3. **认证归属**
   - 确认由原生 App 持有桌面端 token

4. **音乐目标**
   - 确认一期目标是主流播放器优先，不承诺所有播放器完整接管

5. **桥接协议**
   - 确认开发期使用 loopback HTTP 还是本地 socket

## 13. 风险点

### 13.1 风险一：把原组件逐行翻译成 Swift

这是最危险的路线。

当前 Electron 组件已经把窗口控制、接口、认证、提醒、交互、动画、音乐全堆在一个文件里。直接翻译会把问题原封不动带到新架构里。

应对：

- 先拆状态机
- 再拆服务层
- 最后做 UI

### 13.2 风险二：音乐接管目标过高

如果一期目标写成“兼容所有音乐软件”，会导致技术风险失控。

应对：

- Provider 架构
- 主流播放器优先
- 未适配播放器降级

### 13.3 风险三：窗口系统实现过晚

如果太晚才处理 notch、hover、click-through、display change，这些底层问题会反复返工。

应对：

- 提前在 Phase 2 解决窗口壳

### 13.4 风险四：登录状态双写

如果 Electron 和原生各自维护 token，后面很容易状态不同步。

应对：

- 统一 token 持有者
- 统一刷新策略

## 15. 最终建议

本项目的最佳迁移路线不是：

- 继续用 Electron 硬适配 mac 灵动岛
- 或把灵动岛做成没有界面的 helper

而是：

- **把灵动岛做成独立 macOS 原生 App**
- **使用 AppKit + SwiftUI 混合开发**
- **让它成为一个完整、可见、可配置、符合系统习惯的 mac 产品组件**

这样做的优势是：

- 更符合 macOS 用户预期
- 菜单栏、设置页、启动项都天然成立
- 刘海定位和窗口行为更可控
- 音乐接管和系统权限更容易按平台正确建模
- 其他现有业务页面可以继续保留，不需要整站重写

这是当前最稳、也最长期可维护的路线。
